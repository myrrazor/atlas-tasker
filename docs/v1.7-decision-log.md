## D-1702 — Signing And Authority Boundary

- Date: 2026-05-06
- Question: Should signatures grant authority for protected actions?
- Options Considered:
  - Treat valid signatures as sufficient authority.
  - Treat signatures as authenticity only and evaluate authority through governance/permissions.
- Chosen Option: Signatures prove authenticity relative to a key; governance and permission profiles prove authority.
- Why We Chose It: Atlas already has collaborator, membership, gate, and permission models. A key alone cannot prove that the signer was allowed to act.
- Confidence: high
- Revisit Trigger: A future hosted identity-provider integration adds stronger identity and authorization semantics.
- Affected PRs/Files: PR-701 through PR-706, `internal/contracts/security.go`, `internal/contracts/governance.go`

## D-1703 — Key Storage And Trust Sync Boundary

- Date: 2026-05-06
- Question: Which trust and key records sync by default?
- Options Considered:
  - Sync all key and trust material except private keys.
  - Sync public key records and revocations only; keep trust decisions local by default.
- Chosen Option: Public key records and revocations are syncable. Private keys and local trust decisions are local-only unless a future explicit trust-store backup/export flow is approved.
- Why We Chose It: Imported keys and other people's trust decisions should not silently become local authority.
- Confidence: high
- Revisit Trigger: v1.7 adds an explicit signed trust-store backup or team trust ceremony.
- Affected PRs/Files: PR-701, PR-702, PR-703, `.gitignore`, `internal/service/sync_transport.go`

## D-1704 — Verification Is Side-Effect Free By Default

- Date: 2026-05-06
- Question: Should verification commands record events?
- Options Considered:
  - Keep older verify behavior that may append verification events.
  - Make `tracker verify`, status, doctor, replay, reindex, and repair pure by default.
- Chosen Option: Verification is pure by default. Any recorded verification event must be an explicit non-default action.
- Why We Chose It: A check must not become a mutation disguised as inspection, especially for security and audit workflows.
- Confidence: high
- Revisit Trigger: Operators require an explicit compliance workflow for recording verification evidence.
- Affected PRs/Files: PR-701, PR-703, PR-706, `docs/signature-format.md`, `internal/cli/security_commands.go`

## D-1705 — Canonicalization Contract

- Date: 2026-05-06
- Question: How strict should `atlas-c14n-v1` be?
- Options Considered:
  - Use normal JSON/YAML serialization.
  - Freeze a strict typed canonical JSON profile for signed manifests.
- Chosen Option: Use strict canonical JSON bytes: UTF-8, UTC timestamps, exact key sorting, no floats, LF-normalized text, and no volatile fields unless intentionally signed.
- Why We Chose It: Signature verification must be deterministic across workspaces and future implementations.
- Confidence: high
- Revisit Trigger: A signed artifact type needs data that cannot fit this canonical profile.
- Affected PRs/Files: PR-701, PR-702, PR-703, `internal/contracts/canonicalization.go`, `docs/canonicalization.md`

## D-1709 — Public Key Import Does Not Override Local Signing Material

- Date: 2026-05-06
- Question: What should happen when an imported public key record has the same ID as an existing local signing key?
- Options Considered:
  - Let the import overwrite the local public key record.
  - Reject imports that would overwrite local private-backed signing material while allowing idempotent imported-key reimports.
- Chosen Option: Reject public key imports that target an existing `local` public key record; allow already-imported public records to converge through repeated import.
- Why We Chose It: Public key exchange should never weaken or replace local signing authority, but import/sync flows still need to be repeatable.
- Confidence: high
- Revisit Trigger: A future signed trust-store import flow needs a stronger conflict-resolution ceremony.
- Affected PRs/Files: PR-702, `internal/service/security_actions.go`, `internal/service/security_actions_test.go`, `docs/key-management.md`

## D-1710 — Key Identity Is Derived From Public Material

- Date: 2026-05-06
- Question: Should Atlas trust the key ID and fingerprint supplied in a public key file?
- Options Considered:
  - Treat `public_key_id` and `fingerprint` as authoritative metadata.
  - Decode the Ed25519 public bytes, derive fingerprint and key ID, and reject mismatches.
- Chosen Option: Atlas derives and validates key identity from `public_key_material` before storing, importing, or trusting a public key record.
- Why We Chose It: Syncable public records are untrusted input. Trust must bind to the actual cryptographic key, not caller-controlled metadata.
- Confidence: high
- Revisit Trigger: A future multi-algorithm key format changes fingerprint derivation.
- Affected PRs/Files: PR-702, `internal/service/security_store.go`, `internal/service/security_actions_test.go`, `docs/key-management.md`

## D-1711 — Admin Key Scope And Permission Semantics

- Date: 2026-05-06
- Question: How should PR-702 handle admin-scoped keys and platforms without reliable POSIX private-key mode checks?
- Options Considered:
  - Let admin keys fall back to workspace ownership and hard-fail any non-owner-only mode bits.
  - Preserve admin as its own key owner kind and surface unsupported permission checks as explicit warnings.
- Chosen Option: `--scope admin` stores `owner_kind: admin`, and Windows/unsupported permission semantics report `private_key_permissions_unverified` without blocking local signing.
- Why We Chose It: The command must not silently collapse authority boundaries, and unsupported filesystem semantics should be visible without making freshly generated keys unusable on those platforms.
- Confidence: medium
- Revisit Trigger: v1.7 adds a richer admin identity model or encrypted key storage.
- Affected PRs/Files: PR-702, `internal/contracts/security.go`, `internal/service/security_actions.go`, `internal/service/security_store.go`, `docs/key-management.md`

## D-1712 — Signature Context And Local Trust Events

- Date: 2026-05-06
- Question: Should signature verification trust envelope artifact metadata and should local trust ceremonies enter canonical event history?
- Options Considered:
  - Sign only payload bytes and record trust ceremonies in `.tracker/events`.
  - Sign artifact kind/UID plus payload bytes, and keep trust bind/revoke ceremonies in local trust files only.
- Chosen Option: Signatures bind artifact kind, artifact UID, and payload. Trust bind/revoke does not append syncable event-log entries.
- Why We Chose It: Artifact metadata is part of what a verifier is authorizing, and local trust decisions are operator-local authority judgments rather than syncable team claims.
- Confidence: high
- Revisit Trigger: A future signed trust-store export/import flow creates explicit, redacted, operator-approved trust exchange artifacts.
- Affected PRs/Files: PR-702, `internal/service/security_actions.go`, `internal/service/security_actions_test.go`, `docs/signature-format.md`, `docs/key-management.md`

## D-1713 — Expiration Is Evaluated At Use Time

- Date: 2026-05-06
- Question: Is key expiration enforced only by stored key state, or also by the `expires_at` timestamp?
- Options Considered:
  - Require a separate mutation to transition active records to `expired`.
  - Treat `expires_at <= now` as expired for signing and verification, even if the synced record still says `active`.
- Chosen Option: Signing refuses expired timestamps, and verification returns `valid_expired_key` for historical signatures made by expired records.
- Why We Chose It: Imported and synced key records may lag state transitions; expiration is a contract, not advisory metadata.
- Confidence: high
- Revisit Trigger: Governance adds policy-specific grace periods for expiration.
- Affected PRs/Files: PR-702, `internal/service/security_actions.go`, `internal/service/security_actions_test.go`, `docs/key-management.md`

## D-1714 — Trust Binding Attribution Checks

- Date: 2026-05-06
- Question: Can an operator bind any public key to any collaborator ID?
- Options Considered:
  - Allow trust bindings to relabel the key owner at bind time.
  - Require the public key record's collaborator owner to match the requested collaborator ID.
- Chosen Option: `trust bind-key` rejects mismatched owner records, and malformed signature bytes are rejected before any unknown-key validity state is returned.
- Why We Chose It: Trust bindings authorize an existing identity claim; they must not rewrite attribution or let syntactically invalid signatures look valid.
- Confidence: high
- Revisit Trigger: A future trust ceremony supports explicit collaborator key-transfer or delegated signing identities.
- Affected PRs/Files: PR-702, `internal/service/security_actions.go`, `internal/service/security_actions_test.go`, `docs/key-management.md`
