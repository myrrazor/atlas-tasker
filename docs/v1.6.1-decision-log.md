## V161-002
- Date: 2026-03-29
- Question: How should migration safety be modeled during closeout?
- Options Considered:
  - treat migration as a single boolean gate
  - expose a per-entity migration state model with invariants
- Chosen Option: expose `unstamped`, `stamped`, `divergent`, and `unknown` with hard non-regression rules
- Why We Chose It: closeout needs diagnosable state, not a vague pass/fail bit.
- Confidence: high
- Revisit Trigger: a syncable entity class is discovered that cannot be classified with the four-state model.
- Affected PRs/Files: PR-610, sync status, inspect, doctor, migration tests

## V161-004
- Date: 2026-03-29
- Question: How strict should sync-boundary closeout be about remote secrecy and bundle archive objects?
- Options Considered:
  - trust stored remote locations and silently skip unsafe tar entry types
  - reject credential-bearing remotes, redact remote locations on read paths, and fail closed on unsafe archive entry types
- Chosen Option: reject/redact remote secrets and fail closed on unsafe archive objects
- Why We Chose It: closeout is the wrong time to keep permissive parsing around secrets or tar metadata. Silent skipping hides real risk.
- Confidence: high
- Revisit Trigger: Atlas introduces signed remote metadata or a richer transport auth model that changes what counts as sensitive.
- Affected PRs/Files: PR-611, sync transport, bundle verification/import, `.gitignore`, operator docs
