# Atlas Tasker v1.5 Decision Log

## Decision 1 — Pin the train to an immutable v1.4.1 base
- Question: What should v1.5 build on?
- Chosen Option: Build from `codex/pr-412-vcs-faults` at `4f1782e3ef2eaeed06ae0724bd6dc0162a18d940`.
- Why We Chose It: branch names drift; SHAs do not.
- Integration Marker: `v1.5-base-4f1782e`
- Revisit Trigger: only if a newer integrated branch supersedes this base before PR-501 lands.

## Decision 2 — Split provider lifecycle into read then write slices
- Question: Should change sync, review-request, and merge ship in one PR?
- Chosen Option: split the original PR-503 into `PR-503A` and `PR-503B`.
- Why We Chose It: provider drift and authority boundaries are the riskiest part of v1.5.
- Revisit Trigger: none without reducing scope.

## Decision 3 — Split compaction from archival
- Question: Should retention, archival, doctor awareness, and compaction ship together?
- Chosen Option: split the original PR-506 into `PR-506A` and `PR-506B`.
- Why We Chose It: canonical-history safety matters more than bundling convenience.
- Revisit Trigger: none without narrowing compaction further.

## Decision 4 — Claude review fallback is explicit, not blocking
- Question: What happens if Claude Opus 4.5 stalls?
- Chosen Option: retry once, then continue and record the fallback to repo-local gstack skill workflows.
- Why We Chose It: tooling unreliability should not silently block the train.
- Revisit Trigger: none unless Claude availability materially improves.

## Decision 5 — `change import-url` is PR-only in PR-503A
- Question: Should the first provider import command accept any GitHub issue/PR URL?
- Chosen Option: `tracker change import-url` accepts GitHub pull request URLs only in `PR-503A`.
- Why We Chose It: this slice is the change-review lifecycle, not the broader lightweight reference-import story. Pull requests map directly onto `ChangeRef`; issue-reference import remains part of the later import/export scope.
- Revisit Trigger: `PR-505` or later expands lightweight GitHub reference import beyond change-review units.

## Decision 6 — default provider-write authority before permission profiles
- Question: Who may issue provider review-request and merge actions before `PR-504` lands full permission-profile enforcement?
- Chosen Option: `change review-request` is limited to `human:owner` or the run's assigned agent. `change merge` is limited to `human:owner` or the ticket's assigned reviewer.
- Why We Chose It: these defaults stay conservative, map cleanly onto the existing run/reviewer ownership model, and avoid inventing a second temporary permission system before `PR-504`.
- Revisit Trigger: `PR-504` replaces these defaults with permission-profile enforcement and effective-permissions explainers.
