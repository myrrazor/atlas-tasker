# Atlas Tasker v1.6 Decision Log

## V16-001
- Date: 2026-03-28
- Question: What is the immutable integration base for the v1.6 train?
- Options Considered:
  - use the moving `codex/pr-508-release-proof` branch name only
  - pin the current integrated SHA and publish a marker
- Chosen Option: pin the current integrated SHA and publish a marker

- Confidence: high
- Revisit Trigger: the integration base is intentionally rebased or rebuilt before PR-601 lands.

## V16-002
- Date: 2026-03-28
- Question: How do we prevent independently upgraded v1.5 workspaces from minting different sync identities for the same legacy entities?
- Options Considered:
  - lazy random UUIDs on write
  - deterministic UID derivation from immutable legacy anchors plus a pre-sync migration gate
  - syncable migration map after the fact
- Chosen Option: deterministic UID derivation from immutable legacy anchors plus a pre-sync migration gate
- Why We Chose It: it avoids duplicate logical entities on first sync without introducing a second reconciliation system just for upgrades.
- Confidence: high
- Revisit Trigger: legacy entities are discovered that do not have stable immutable anchors for deterministic derivation.
- Affected PRs/Files: PR-601 through PR-608, contracts, codecs, sync transport, tests

## V16-003
- Date: 2026-03-28
- Question: Is `logical_clock` a real logical clock or just a local sequence number?
- Options Considered:
  - keep the name but use a simple local counter
  - rename it to a workspace sequence
  - define it as a Lamport-style clock
- Chosen Option: define it as a Lamport-style clock
- Why We Chose It: the field needs honest distributed semantics because v1.6 orders and reconciles multi-workspace event streams.
- Confidence: high
- Revisit Trigger: causality requirements grow beyond Lamport semantics and require a richer clock model.
- Affected PRs/Files: PR-601 through PR-608, contracts, event model, sync reconciliation, tests
