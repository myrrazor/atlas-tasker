# Atlas Tasker v1.3 Decision Log

## V13-001
- Date: 2026-03-23
- Question: What branch model should v1.3 use when the repo does not have `dev` or `testing` branches?
- Options Considered:
  - create new long-lived branches just for v1.3
  - continue the existing stacked `codex/pr-*` branch model
- Chosen Option: continue the existing stacked `codex/pr-*` branch model
- Why We Chose It: the repo already runs on stacked branches, and inventing a second branch strategy mid-train would add process churn without improving code safety.
- Confidence: high
- Revisit Trigger: the repo adds `dev`/`testing` and the team migrates all active work to that model.
- Affected PRs/Files: PR-301..PR-309 planning and branch strategy.

## V13-002
- Date: 2026-03-23
- Question: Should workspace event allocation use a separate coordination mechanism from write locking?
- Options Considered:
  - separate event ID allocator
  - allocate event IDs under the same workspace write lock as the mutation
- Chosen Option: allocate event IDs under the same workspace write lock as the mutation
- Why We Chose It: one coordination mechanism is simpler to reason about, easier to test, and keeps concurrent local writers deterministic.
- Confidence: high
- Revisit Trigger: Atlas adds multi-machine writers.
- Affected PRs/Files: PR-301 locking and event allocation.

## V13-003
- Date: 2026-03-23
- Question: Should arbitrary local command execution ship in the base automation slice?
- Options Considered:
  - include local command execution in PR-303
  - defer it and ship safe built-in actions only
- Chosen Option: defer it and ship safe built-in actions only
- Why We Chose It: local command execution expands the safety and testing surface too much for the base automation slice.
- Confidence: high
- Revisit Trigger: a later v1.3.x or v1.4 train is ready to ship a guarded experimental execution model.
- Affected PRs/Files: PR-303 automation scope.

## V13-004
- Date: 2026-03-23
- Question: How should Atlas detect and repair partially committed local mutations?
- Options Considered:
  - best-effort rebuilds with no explicit mutation journal
  - file-backed mutation journal entries that survive crashes until repair clears them
- Chosen Option: file-backed mutation journal entries under `.tracker/mutations`
- Why We Chose It: the journal keeps pending local writes explicit, lets repair replay a missing event conservatively, and gives operators something concrete to inspect when a mutation stops mid-flight.
- Confidence: medium-high
- Revisit Trigger: Atlas needs multi-workspace or remote mutation recovery.
- Affected PRs/Files: PR-302 durability, `internal/service/journal.go`, `docs/storage-transaction-model.md`.

## V13-005
- Date: 2026-03-23
- Question: How should full projection rebuilds avoid leaving the main SQLite index half-rewritten?
- Options Considered:
  - keep rebuilding the live database in place
  - rebuild into a temp database and swap it into place when complete
- Chosen Option: rebuild into a temp database and swap it into place for full rebuilds
- Why We Chose It: rebuild-then-swap turns projection repair into an all-or-nothing step and keeps a broken rebuild from trashing the main read model.
- Confidence: high
- Revisit Trigger: Atlas adds very large workspaces where full projection swap becomes too expensive.
- Affected PRs/Files: PR-302 durability and repair, `internal/storage/sqlite/store.go`.

## V13-006
- Date: 2026-03-23
- Question: What should automation rule files do when `enabled` is omitted?
- Options Considered:
  - default omitted rules to disabled
  - default omitted rules to enabled and require explicit `enabled = false` to disable them
- Chosen Option: default omitted rules to enabled
- Why We Chose It: the CLI already writes explicit enabled state, and default-enabled files are less surprising for hand-authored rules.
- Confidence: medium-high
- Revisit Trigger: operators start relying on checked-in but intentionally dormant rules.
- Affected PRs/Files: PR-303 automation store loading, `internal/service/automation_store.go`.

## V13-008
- Date: 2026-03-23
- Question: How should notifier failures be tracked without rolling back committed mutations?
- Options Considered:
  - return the first sink error and drop the rest
  - record every sink attempt in a delivery log and final failures in a dead-letter log
- Chosen Option: record every sink attempt in a delivery log and final failures in a dead-letter log
- Why We Chose It: v1.3 needs best-effort post-commit delivery, but operators still need a durable trail when webhook or file sinks fail.
- Confidence: high
- Revisit Trigger: Atlas adds background retry workers or subscriptions with per-recipient state.
- Affected PRs/Files: PR-304 notifier transport, `internal/service/notify.go`, `docs/json-contracts.md`.

## V13-009
- Date: 2026-03-23
- Question: How should Atlas suggest local git branches from ticket context?
- Options Considered:
  - infer branches from current repo naming conventions
  - emit a stable Atlas branch shape and treat it as advisory only
- Chosen Option: emit `ticket/<ticket-id>-<slug>` as an advisory branch name
- Why We Chose It: a stable branch shape makes queue/detail hints predictable without turning Git into a second workflow source of truth.
- Confidence: medium-high
- Revisit Trigger: teams standardize on a different branch convention that needs to be configurable.
- Affected PRs/Files: PR-305 local git integration, `internal/service/scm.go`.

## V13-010
- Date: 2026-03-24
- Question: How should Atlas persist saved views without turning them into config-only global state?
- Options Considered:
  - extend `config.toml` with named views
  - store each saved view as its own local tracker artifact under `.tracker/views`
- Chosen Option: store each saved view as a `.toml` file under `.tracker/views`
- Why We Chose It: per-view files match the existing automation/template pattern, keep diffs readable, and give later watcher/TUI work a stable local target without overloading `config.toml`.
- Confidence: high
- Revisit Trigger: Atlas adds team-shared or remote-synced view definitions.
- Affected PRs/Files: PR-307A saved views, `internal/service/view_store.go`, `internal/contracts/views.go`.

## V13-011
- Date: 2026-03-24
- Question: How should watcher rules affect notification delivery once subscriptions exist?
- Options Considered:
  - keep always-on v1.2 notify behavior and treat watchers as metadata only
  - make watcher matches additive on top of the old notify-worthy event list
  - switch to watcher-driven delivery whenever at least one watcher exists in the workspace
- Chosen Option: keep the old notify-worthy event list as the baseline, and add watcher-driven recipients on top of it
- Why We Chose It: watchers should make operators more informed, not silently suppress existing high-signal notifications for unrelated work.
- Confidence: high
- Revisit Trigger: Atlas adds per-recipient delivery state or background retry workers.
- Affected PRs/Files: PR-307B subscriptions, `internal/service/notify.go`, `internal/service/subscriptions.go`.

## V13-012
- Date: 2026-03-24
- Question: How should bulk operations record traceability and partial failures without inventing a second mutation path?
- Options Considered:
  - build a separate batch mutation subsystem with its own event format
  - reuse normal per-ticket mutations and stamp a shared `metadata.batch_id` across the batch
- Chosen Option: reuse normal per-ticket mutations and propagate one shared `batch_id`
- Why We Chose It: bulk work should stay debuggable through the same history and repair surfaces as single-ticket mutations, while still giving operators one preview and one aggregate result to reason about.
- Confidence: high
- Revisit Trigger: Atlas adds true transactional multi-ticket commits or server-coordinated batch workflows.
- Affected PRs/Files: PR-307C bulk operations, `internal/service/bulk.go`, `docs/json-contracts.md`.

## V13-013
- Date: 2026-03-24
- Question: How should the v1.3 TUI expose bulk work and the new operator surfaces without duplicating service logic?
- Options Considered:
  - add one-off TUI-only flows for notifications, views, automation, and bulk work
  - keep the TUI as a thin client over shared queries, use dedicated tabs for the new read surfaces, and make bulk work a preview-then-apply flow
- Chosen Option: add Inbox, Views, and Ops tabs on top of shared query methods, and expose bulk work as `b` preview then `y` apply
- Why We Chose It: the TUI stays additive, bulk work preserves explicit confirmation semantics, and the UI can show new operator context without becoming a second business-logic layer.
- Confidence: high
- Revisit Trigger: Atlas adds richer modal workflows or a more structured command palette state machine.
- Affected PRs/Files: PR-308 TUI expansion, `internal/tui/app.go`, `internal/tui/mutations.go`, `internal/service/query.go`.
