# Atlas Tasker v1.3 Decision Log

## V13-001
- Date: 2026-03-23
- Question: What branch model should v1.3 use when the repo does not have `dev` or `testing` branches?
- Options Considered:
  - create new long-lived branches just for v1.3
  - continue the existing stacked `codex/pr-*` branch model
- Chosen Option: continue the existing stacked `codex/pr-*` branch model
- Why We Chose It: the repo already runs on stacked branches, and inventing a second branch strategy mid-train would add process churn without improving code safety.
- Confidence: high
- Revisit Trigger: the repo adds `dev`/`testing` and the team migrates all active work to that model.
- Affected PRs/Files: PR-301..PR-309 planning and branch strategy.

## V13-002
- Date: 2026-03-23
- Question: Should workspace event allocation use a separate coordination mechanism from write locking?
- Options Considered:
  - separate event ID allocator
  - allocate event IDs under the same workspace write lock as the mutation
- Chosen Option: allocate event IDs under the same workspace write lock as the mutation
- Why We Chose It: one coordination mechanism is simpler to reason about, easier to test, and keeps concurrent local writers deterministic.
- Confidence: high
- Revisit Trigger: Atlas adds multi-machine writers.
- Affected PRs/Files: PR-301 locking and event allocation.

## V13-003
- Date: 2026-03-23
- Question: Should arbitrary local command execution ship in the base automation slice?
- Options Considered:
  - include local command execution in PR-303
  - defer it and ship safe built-in actions only
- Chosen Option: defer it and ship safe built-in actions only
- Why We Chose It: local command execution expands the safety and testing surface too much for the base automation slice.
- Confidence: high
- Revisit Trigger: a later v1.3.x or v1.4 train is ready to ship a guarded experimental execution model.
- Affected PRs/Files: PR-303 automation scope.
