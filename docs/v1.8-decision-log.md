# Atlas Tasker v1.8 Decision Log

This log records v1.8 release-candidate polish decisions. v1.8 does not redesign Atlas. It makes the product public-readable, cleaner to operate, easier for coding agents to use, and better proven as a release candidate.

- Date: 2026-05-07
- Question: Where should v1.8 start while the v1.7 stack is still open?
- Options Considered:
  - Wait until PRs #68-#76 merge into `main`, then start PR-801 from updated `main`.
  - Start from current `main`, treating v1.7/v1.7.1 as missing blockers.
  - Stack PR-801 on `codex/pr-709-v171-release-signoff` at `5a75aec336311abd1844c8578342e7da0af5388e`, then retarget or rebase after the v1.7 stack merges.

## D-1802 — v1.8 Is Release Polish, Not A New Subsystem Train

- Date: 2026-05-07
- Question: What kind of work belongs in v1.8?
- Options Considered:
  - Add another product subsystem before public launch.
  - Limit v1.8 to public release polish, docs, terminal/TUI polish, agent UX, demos, release validation, and security review.
- Chosen Option: v1.8 is release-candidate polish only.
- Why We Chose It: Atlas already has the core local orchestration, governance, signing, redaction, audit, backup, goal, and MCP surfaces. The remaining first-release gap is trust from humans: readable docs, crisp examples, clean terminal output, and reproducible release proof.

- Revisit Trigger: A release-blocking bug requires a narrow product fix.
- Affected PRs/Files: PR-801 through PR-809

## D-1804 — Public Claims Carry The v1.7 No-Ship Block Forward

## D-1805 — Security Wording Stays Precise And Limited

- Date: 2026-05-07
- Question: How should public docs describe Atlas security?
- Options Considered:
  - Use broad launch language such as secure, enterprise-ready, encrypted, or identity-backed.
  - State exact claims: trusted local-key signature verification, app-level governance, structured redaction for Atlas-owned data, signed audit packets, and side-effect-safe restore planning.
- Chosen Option: Use exact security claims and repeat limitations wherever release trust is discussed.
- Why We Chose It: Atlas is local-first software, not an OS sandbox, hosted identity provider, encrypted vault, or DLP system. Overclaiming security would be worse than saying the true boundary plainly.
- Confidence: high
- Revisit Trigger: Atlas adds encrypted storage, hosted identity, or OS sandbox support.
- Affected PRs/Files: PR-802, PR-803, PR-807, PR-809, `docs/security-limitations.md`

## D-1806 — Docs Architecture Separates Public Docs From Historical Plans

- Date: 2026-05-07

- Options Considered:

- Revisit Trigger: The docs site moves from GitHub Markdown to a generated website.

## D-1808 — Review Tooling Is Required But Not A Hard Blocker When Unavailable

- Date: 2026-05-07
- Question: How should the v1.8 train treat GStack/Codex review tooling failures?
- Options Considered:
  - Block each PR indefinitely until every model review runs.
  - Attempt the requested pre/post reviews, record failures, and keep deterministic local gates mandatory.
- Chosen Option: Attempt GStack/Codex reviews and record unavailable tooling; deterministic tests remain the release gate.
- Why We Chose It: Model reviews are useful, but release proof must be reproducible. Tooling availability should not stop a docs or polish PR when tests and manual review can still prove the change.
- Confidence: high
- Revisit Trigger: A required security review finds a high-confidence blocking issue.
- Affected PRs/Files: PR-801 through PR-809, `docs/v1.8-decision-log.md`
