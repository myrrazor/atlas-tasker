# Atlas Tasker v1.4 Decision Log

- Options Considered: upstream `main`; local integrated `codex/pr-314-stabilization-proof`
- Chosen Option: local integrated `codex/pr-314-stabilization-proof`

## V14-002
- Date: 2026-03-25
- Question: How should replay, repair, and reindex treat v1.4 orchestration side effects?
- Options Considered: recreate side effects from events; projection/journal only reconciliation
- Chosen Option: projection/journal only reconciliation
- Why We Chose It: worktrees, runtime files, notifier delivery, and automation are live side effects and must not be replayed.
- Confidence: high
- Revisit Trigger: a future server mode introduces explicit durable side-effect orchestration.
- Affected PRs/Files: PR-401, PR-403, PR-405, PR-406, PR-407, PR-409

## V14-003
- Date: 2026-03-25
- Question: Where should provider-specific context live?
- Options Considered: rewrite project-level guidance every run; keep project guidance stable and write run-scoped runtime artifacts
- Chosen Option: stable project guidance plus run-scoped runtime artifacts
- Why We Chose It: Codex and Claude consume durable repo instructions well, while run-specific context changes frequently and belongs in `.tracker/runtime/<run-id>/`.
- Confidence: high
- Revisit Trigger: providers add first-class run-scoped root guidance support.
- Affected PRs/Files: PR-401, PR-407, PR-408

## V14-004
- Date: 2026-03-25
- Question: How should workspace-scoped agent mutations appear in the event log?
- Options Considered: no events for agent changes; reuse the event log with a reserved workspace pseudo-project key
- Chosen Option: use reserved project key `_workspace` for workspace-scoped agent events
- Why We Chose It: it preserves append-only mutation history without redesigning the event store around non-project scopes.
- Confidence: medium
- Revisit Trigger: if future workspace-scoped entities need a first-class non-project event partition.
- Affected PRs/Files: PR-402, internal/service/action.go, internal/contracts/events.go

## V14-005
- Date: 2026-03-25
- Question: What should the first eligibility ranking rule prefer?
- Options Considered: arbitrary file order; deterministic routing weight then lexical agent id
- Chosen Option: deterministic routing weight then lexical agent id after eligibility filtering
- Why We Chose It: stable ranking is easy to reason about, test, and reuse later in dispatch without adding premature complexity.
- Confidence: high
- Revisit Trigger: PR-404 needs richer ranking signals from runbook stage affinity or active run counts.
- Affected PRs/Files: PR-402, internal/service/query.go

## V14-006
- Date: 2026-03-25
- Question: What counts as a dirty source repo for worktree dispatch safety?
- Options Considered: block on any repo change including Atlas metadata; ignore Atlas-managed workspace files and block only on non-Atlas repo changes
- Chosen Option: ignore Atlas-managed workspace files when enforcing `require_clean_main`
- Why We Chose It: Atlas necessarily writes tickets, projects, and `.tracker` metadata inside the repo, so treating those files as dispatch blockers would make worktree dispatch unusable in normal workspaces.
- Confidence: high
- Revisit Trigger: Atlas gains a separate metadata root outside the source repo or a more explicit ignore contract.
- Affected PRs/Files: PR-403, internal/service/worktree.go

## V14-007
- Date: 2026-03-25
- Question: How should dispatch order canonical writes versus worktree creation?
- Options Considered: create the worktree before writing the run snapshot; persist the run snapshot first and roll back on worktree/runtime setup failure
- Chosen Option: persist the run snapshot first, then create the worktree/runtime dir inside the locked canonical write path with explicit rollback on failure
- Why We Chose It: it keeps the durable run record ahead of live side effects without leaving half-committed canonical state behind if worktree setup fails.
- Confidence: medium
- Revisit Trigger: mutation journals gain explicit sub-stage recovery for run/worktree setup.
- Affected PRs/Files: PR-403, internal/service/run_actions.go

## V14-008
- Date: 2026-03-25
- Question: In what order should dispatch resolve a runbook?
- Options Considered: project defaults first; agent defaults first; explicit ticket override, then agent default, then project mapping, then built-in fallback
- Chosen Option: explicit ticket override, then agent default, then project mapping, then built-in fallback
- Why We Chose It: ticket-local intent should win, agent defaults keep routing practical, project mappings keep sensible workspace defaults, and built-ins prevent dead ends.
- Confidence: high
- Revisit Trigger: runbook inheritance becomes stage-aware or multi-runbook dispatch is introduced.
- Affected PRs/Files: PR-404, internal/service/dispatch.go, internal/service/runbook_store.go

## V14-009
- Date: 2026-03-25
- Question: How should bulk dispatch handle items that are no longer eligible at apply time?
- Options Considered: trust the original saved-view expansion; re-check eligibility for each ticket right before dispatch
- Chosen Option: re-check eligibility for each ticket at preview and apply time
- Why We Chose It: active runs, gates, and other dispatch blockers can change between view expansion and execution, and the live command should report those conflicts instead of forcing stale decisions through.
- Confidence: high
- Revisit Trigger: Atlas adds snapshot-locked multi-ticket orchestration transactions.
- Affected PRs/Files: PR-404, internal/cli/dispatch.go

## V14-010
- Date: 2026-03-25
- Question: How should evidence lifecycle work in v1.4?
- Options Considered: allow edits/deletes in place; keep evidence immutable and model changes as new entries that may supersede earlier ones
- Chosen Option: evidence is immutable and may only be superseded by newer evidence entries
- Why We Chose It: evidence is part of the operator audit trail, and immutable entries keep replay, export, and handoff generation deterministic.
- Confidence: high
- Revisit Trigger: Atlas adds explicit redaction workflows for sensitive evidence.
- Affected PRs/Files: PR-405, internal/service/evidence_actions.go, internal/service/evidence_handoff_store.go

## V14-011
- Date: 2026-03-25
- Question: Where should evidence artifacts and handoff exports live?
- Options Considered: keep external file references only; copy artifacts into run-scoped tracker storage and render handoff exports from stored packets
- Chosen Option: copy evidence artifacts into `.tracker/evidence/<run-id>/` and render deterministic handoff markdown from the stored packet
- Why We Chose It: bundling artifacts under tracker storage keeps evidence portable with the workspace and makes handoff export deterministic even after the source worktree changes.
- Confidence: high
- Revisit Trigger: Atlas adds large-artifact offloading or remote evidence storage.
- Affected PRs/Files: PR-405, internal/service/evidence_actions.go, internal/cli/evidence.go

## V14-012
- Date: 2026-03-25
- Question: When should PR-406 open gates?
- Options Considered: separate explicit `gate open` mutations only; automatically open required gates during `run handoff`
- Chosen Option: `run handoff` opens any required runbook gates for the run and can add an explicit `--next-gate`
- Why We Chose It: handoff generation is the point where operator intent becomes concrete, and opening gates there keeps the handoff packet, ticket gate state, and inbox in one atomic mutation path.
- Confidence: medium
- Revisit Trigger: Atlas adds a first-class stage promotion command or explicit gate templating workflows.
- Affected PRs/Files: PR-406, internal/service/evidence_actions.go, internal/service/gate_actions.go

## V14-013
- Date: 2026-03-25
- Question: How should gate decisions affect run state?
- Options Considered: gate decisions never touch runs; rejected gates return the run to `active` and resolved gates relax the run to `handoff_ready` when no other gates remain
- Chosen Option: rejected gates send the linked run back to `active`, while the last approved or waived run-scoped gate relaxes the run to `handoff_ready`
- Why We Chose It: it gives operators a clear retry loop without auto-completing work and keeps the inbox transition obvious after a decision.
- Confidence: medium
- Revisit Trigger: Atlas adds richer stage progression beyond `handoff_ready`, `awaiting_review`, and `awaiting_owner`.
- Affected PRs/Files: PR-406, internal/service/gate_actions.go, internal/service/run_actions.go

## V14-014
- Date: 2026-03-25
- Question: How should runtime launch artifacts behave when operators regenerate them?
- Options Considered: rewrite runtime files on every `run launch`; keep `run launch` idempotent and require explicit refresh to rewrite existing files
- Chosen Option: `run launch` is idempotent by default and only rewrites existing files when `--refresh` is set
- Why We Chose It: operators need stable artifact paths they can hand to Codex or Claude without accidental churn, while still keeping an explicit escape hatch for stale launch material.
- Confidence: high
- Revisit Trigger: runtime manifests gain durable version history or staged regeneration semantics.
- Affected PRs/Files: PR-407, internal/service/run_launch.go, internal/integrations/runtime.go, internal/cli/run.go
