# Atlas Tasker v1.4 Decision Log

## V14-001
- Date: 2026-03-25
- Question: What branch is the real implementation base for v1.4?
- Options Considered: upstream `main`; local integrated `codex/pr-314-stabilization-proof`
- Chosen Option: local integrated `codex/pr-314-stabilization-proof`
- Why We Chose It: upstream is still behind the merged v1.2-v1.3.1 stack, so building on main would discard real repo state.
- Confidence: high
- Revisit Trigger: upstream catches up and the v1.4 stack is retargeted.
- Affected PRs/Files: PR-401 through PR-409

## V14-002
- Date: 2026-03-25
- Question: How should replay, repair, and reindex treat v1.4 orchestration side effects?
- Options Considered: recreate side effects from events; projection/journal only reconciliation
- Chosen Option: projection/journal only reconciliation
- Why We Chose It: worktrees, runtime files, notifier delivery, and automation are live side effects and must not be replayed.
- Confidence: high
- Revisit Trigger: a future server mode introduces explicit durable side-effect orchestration.
- Affected PRs/Files: PR-401, PR-403, PR-405, PR-406, PR-407, PR-409

## V14-003
- Date: 2026-03-25
- Question: Where should provider-specific context live?
- Options Considered: rewrite project-level guidance every run; keep project guidance stable and write run-scoped runtime artifacts
- Chosen Option: stable project guidance plus run-scoped runtime artifacts
- Why We Chose It: Codex and Claude consume durable repo instructions well, while run-specific context changes frequently and belongs in `.tracker/runtime/<run-id>/`.
- Confidence: high
- Revisit Trigger: providers add first-class run-scoped root guidance support.
- Affected PRs/Files: PR-401, PR-407, PR-408

## V14-004
- Date: 2026-03-25
- Question: How should workspace-scoped agent mutations appear in the event log?
- Options Considered: no events for agent changes; reuse the event log with a reserved workspace pseudo-project key
- Chosen Option: use reserved project key `_workspace` for workspace-scoped agent events
- Why We Chose It: it preserves append-only mutation history without redesigning the event store around non-project scopes.
- Confidence: medium
- Revisit Trigger: if future workspace-scoped entities need a first-class non-project event partition.
- Affected PRs/Files: PR-402, internal/service/action.go, internal/contracts/events.go

## V14-005
- Date: 2026-03-25
- Question: What should the first eligibility ranking rule prefer?
- Options Considered: arbitrary file order; deterministic routing weight then lexical agent id
- Chosen Option: deterministic routing weight then lexical agent id after eligibility filtering
- Why We Chose It: stable ranking is easy to reason about, test, and reuse later in dispatch without adding premature complexity.
- Confidence: high
- Revisit Trigger: PR-404 needs richer ranking signals from runbook stage affinity or active run counts.
- Affected PRs/Files: PR-402, internal/service/query.go

## V14-006
- Date: 2026-03-25
- Question: What counts as a dirty source repo for worktree dispatch safety?
- Options Considered: block on any repo change including Atlas metadata; ignore Atlas-managed workspace files and block only on non-Atlas repo changes
- Chosen Option: ignore Atlas-managed workspace files when enforcing `require_clean_main`
- Why We Chose It: Atlas necessarily writes tickets, projects, and `.tracker` metadata inside the repo, so treating those files as dispatch blockers would make worktree dispatch unusable in normal workspaces.
- Confidence: high
- Revisit Trigger: Atlas gains a separate metadata root outside the source repo or a more explicit ignore contract.
- Affected PRs/Files: PR-403, internal/service/worktree.go

## V14-007
- Date: 2026-03-25
- Question: How should dispatch order canonical writes versus worktree creation?
- Options Considered: create the worktree before writing the run snapshot; persist the run snapshot first and roll back on worktree/runtime setup failure
- Chosen Option: persist the run snapshot first, then create the worktree/runtime dir inside the locked canonical write path with explicit rollback on failure
- Why We Chose It: it keeps the durable run record ahead of live side effects without leaving half-committed canonical state behind if worktree setup fails.
- Confidence: medium
- Revisit Trigger: mutation journals gain explicit sub-stage recovery for run/worktree setup.
- Affected PRs/Files: PR-403, internal/service/run_actions.go
